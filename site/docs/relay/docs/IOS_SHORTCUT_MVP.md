# iOS 快捷指令 MVP（Action Button 语音输入）

不写一行原生 iOS 代码，用「快捷指令 + Action Button」复刻 mac 端 Phase 1 的核心管线：

| mac 端 | iOS 快捷指令对应 |
|---|---|
| Option 键触发录音（AudioRecorder） | Action Button 触发「录制音频」动作 |
| 音频 base64 + 场景化 system prompt → 一次多模态调用（AIAudioProvider） | 「获取 URL 内容」POST Gemini 或阿里 Qwen Omni，音频 base64 + 精简版 system prompt |
| TextInjector 注入光标处 | 结果进剪贴板 + 通知，手动粘贴（iOS 系统限制，无解） |
| SceneManager 场景自适应 | 不支持（拿不到前台 app），用通用 prompt |
| QA / Agent / Web Search / 短期记忆 | 全部砍掉 |

保留的核心能力：**意图式输入**（"跟他说…"、"翻译成英文…"、"改得正式点…" 这类元指令剥离与执行）、口语噪音清理、人称转换、[NO_SPEECH] 噪音闸门、个人词典纠错。

## 选方案：Gemini 还是阿里 Qwen Omni

| | 方案 A：OpenRouter（Gemini） | 方案 B：阿里 Qwen Omni（推荐国内使用） |
|---|---|---|
| 模型 | `google/gemini-3.5-flash`（mac 端 OpenRouter 路线默认） | `qwen3.5-omni-flash`（mac 端 Qwen 路线默认） |
| 国内直连 | ❌ 需代理 | ✅ DashScope 国内可直连 |
| key | 复用 mac `Secrets.swift` 里的 `openRouter` | 复用 mac `Secrets.swift` 里的 `aliyunQwen` |
| m4a 支持 | ✅ 官方支持 M4A | ✅ 官方支持 M4A |
| 响应解析 | 非流式 JSON，逐层取值，简单 | **只支持流式输出**（SSE），要先把文件转文本再正则拼接，多 7 个动作 |
| 特殊参数 | `reasoning: {effort: minimal}`（Gemini 3.x 必配 thinking 档位；不传 temperature） | `stream: true`（协议强制）、`modalities: ["text"]` |

手机在国内用选方案 B（OpenRouter 不挂代理基本不通）；海外或常驻代理选方案 A（解析简单得多）。

## 前置准备

1. API key：按上表准备其一。
2. iPhone 15 Pro 及以上（有 Action Button），或用锁屏/主屏幕图标触发也行。

## 快捷指令搭建 — 方案 A：OpenRouter + Gemini（共约 12 个动作）

新建快捷指令，命名如「语音输入」，按顺序添加（方案 B 只有 ④ 之后不同，见下一节）：

### ① 录制音频

- 动作：**录制音频**（媒体类）
- 开始录制：**立即**（按下 Action Button 就是要说话，别再多点一下）
- 结束录制：**轻点时**（说完轻点停止）

### ② Base64 编码

- 动作：**Base64 编码**，输入 = 上一步的「录制的音频」
- ⚠️ 把「换行」选项设为**无**（默认每 76 字符插换行，会污染请求体）

### ③ 文本（system prompt）

- 动作：**文本**，把本文末尾「精简版 System Prompt」整段粘贴进去。
- 词典段落按需修改（你的名字、邮箱、常用专有名词）。以后调 prompt 只改这个动作。

### ④ 获取 URL 内容（核心请求）

- 动作：**获取 URL 内容**
- URL：`https://openrouter.ai/api/v1/chat/completions`
- 方法：**POST**
- 标头：键 `Authorization`，值 `Bearer sk-or-v1-你的key`（Bearer + 空格 + key 整串填在值里；Content-Type 不用加，请求体选 JSON 后自动带）
- 请求体：**JSON**（注：界面里 Array 叫**列表**），按以下结构搭：

```text
model       （文本）= google/gemini-3.5-flash
messages    （列表，2 项）
├─ （字典）role（文本）= system，content（文本）= ③ 的「文本」变量
└─ （字典）role（文本）= user
   └─ content            （列表，1 项）
      └─ （字典）
         ├─ type         （文本）= input_audio
         └─ input_audio  （字典）
            ├─ data      （文本）= ② 的「Base64 编码」变量（无前缀，直接放变量）
            └─ format    （文本）= m4a
max_tokens  （数字）= 4096
reasoning   （字典）
└─ effort   （文本）= minimal
```

注意两点（与 mac 端 `OpenRouterModelConfig` 的处理一致）：

- **`reasoning.effort` 必须配**——Gemini 3.x 是思考模型，不限档位会把时间和 token 烧在推理上；语音转写场景 `minimal` 足够（mac 端的可选档位就是 minimal/low/medium/high）。
- **不要传 `temperature`**——Gemini 3.x 路线 mac 端明确省略采样参数（`shouldOmitSamplingParameters`），照抄。

等价的最终请求体（仅供核对，不用手写）：

```json
{
  "model": "google/gemini-3.5-flash",
  "messages": [
    { "role": "system", "content": "<system prompt>" },
    { "role": "user", "content": [
      { "type": "input_audio", "input_audio": { "data": "<base64 m4a>", "format": "m4a" } }
    ] }
  ],
  "max_tokens": 4096,
  "reasoning": { "effort": "minimal" }
}
```

### ⑤–⑨ 提取返回文本

响应是标准 JSON（Content-Type `application/json`，快捷指令直接识别为词典，**没有**方案 B 那个"未知文件"问题），取 `choices[0].message.content`：

5. **获取词典值**：键 `choices`（输入 = URL 内容结果）
6. **从列表获取项目**：首个项目
7. **获取词典值**：键 `message`
8. **获取词典值**：键 `content`
9. （备用位，不需要——方案 A 到 ⑧ 就拿到最终文本了）

### ⑩–⑫ 结果处理

10. **如果**：上一步的文本 **包含** `NO_SPEECH`
    - **显示通知**：「未识别到有效语音」
11. **否则**：
    - **拷贝到剪贴板**：该文本
    - **显示通知**：标题「已复制」，正文 = 该文本
12. **结束如果**

## 快捷指令搭建 — 方案 B：阿里 Qwen Omni

①②③（录音、Base64、prompt 文本）与方案 A 完全相同，从 ④ 开始替换：

### ④ 获取 URL 内容

- URL：`https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`（与 mac 端 Direct 模式同一端点，见 `RelayAPIClient.directProviderURL`）
- 方法：**POST**
- 标头：`Authorization` = `Bearer sk-你的DashScope key`（即 mac `Secrets.aliyunQwen`）
- 请求体：**JSON**：

```text
model       （文本）= qwen3.5-omni-flash
messages    （数组，2 项）
├─ （字典）role = system，content（文本）= ③ 的「文本」变量
└─ （字典）role = user
   └─ content            （数组，1 项）
      └─ （字典）
         ├─ type         （文本）= input_audio
         └─ input_audio  （字典）
            ├─ data      （文本）= data:;base64,［② 的 Base64 变量］ ← 前缀手动输入，变量紧跟其后
            └─ format    （文本）= m4a
modalities  （数组，1 项文本）= text
stream      （布尔）= 真
```

⚠️ `stream` 必须为真——**Qwen Omni 系列只支持流式输出**，传 false 会直接报错（mac 端 `AliyunQwenAI.swift` 同样硬编码 `stream: true`）。

### ⑤–⑪ 从 SSE 流里拼出文本

快捷指令不会增量消费 SSE，「获取 URL 内容」会等连接结束后返回整段文本（一堆 `data: {...}` 行），生成耗时和非流式一样，只是解析方式不同。这和 mac/Android 端的行为完全一致——两端也只是逐行读 SSE、把 `delta.content` 拼成完整文本后一次性使用（见 `RelayAPIClient.stream` / Android `AliyunQwenAI.sendStreamingRequest`），`stream: true` 纯粹是 DashScope 对 omni 模型的协议要求，应用层从来不是真流式。

⚠️ **关键前置：响应是"文件"不是文本。** SSE 响应的 `Content-Type: text/event-stream` 快捷指令不认识，④ 的输出会被包装成未知类型的文件对象。直接拿去「匹配文本」会得到 0 个匹配且**不报任何错**，后续全链路静默空转，最后"成功"复制一个空字符串。必须先用 ⑤⑥ 两步把它转成文本：

5. **设定名称**（Set Name）：输入 = ④ 的「URL 的内容」，名称填 `response.txt`（补上 .txt 后缀，快捷指令才肯按文本解码）
6. **获取文本**（从输入获取文本）：输入 = ⑤ 的「已重新命名的项目」。输出才是真正的 SSE 文本（注意：如果跳过 ⑤ 直接获取文本，拿到的会是文件名 "completions"）
7. **匹配文本**（输入 = ⑥ 的「文本」）：

   ```text
   (?<!reasoning_)\x{22}content\x{22}:\x{22}((?:[^\x{22}\\]|\\.)*)\x{22}
   ```

   `\x{22}` 是引号字符的正则写法——故意不写字面引号，避免 iOS 智能标点把直引号换成弯引号（正则会因此一个都匹配不到）。负向后顾排除 thinking 模式下的 `reasoning_content` 字段。「匹配文本」动作没有正则开关，它永远按正则解释。
8. **获取匹配文本的组**（在匹配中获取索引处的组）：组索引 **1** → 得到内容片段列表
9. **合并文本**：结合方式选**自定**，分隔符留**空**。⚠️ 输入必须手动选 ⑧ 的**「匹配文本的组」**——此时流程里有多个叫"文本"的变量（③ 的 prompt、⑥ 的输出），接错成 ③ 会导致输出整个 system prompt，且因 prompt 含 [NO_SPEECH] 字样而永远走进"未识别"分支
10. **替换文本**：查找 `\n`（正则开关**关**，按字面找"反斜杠+n"两个字符），替换为真实换行（替换框里直接按回车；按不了就建一个只含一个回车的「文本」动作当变量插入）
11. **替换文本**：查找 `\"`（正则开关关），替换为 `"`

之后接方案 A 的结果处理（NO_SPEECH 判断、剪贴板、通知），不变。

### 方案 B 故障排查

| 现象 | 处理 |
|---|---|
| 全链路无报错但剪贴板是空的 | 八成是漏了 ⑤⑥ 的文件转文本（见上方关键前置）；其次检查「拷贝到剪贴板」的内容是否手动选了最终文本变量——分支内的动作不会自动接上游输出 |
| 「显示结果」调试时只看到一个 ？文件 | 同上，④ 的输出是未知类型文件，接 ⑥ 之后的变量看 |
| 「获取文本」输出是 "completions" | 拿到的是文件名——缺了 ⑤ 设定名称那步 |
| 匹配数为 0 但响应正常 | 正则里的引号被智能标点换成了弯引号（手敲必中），改用上面 `\x{22}` 版正则 |
| 报错提示不支持非流式 | 检查 `stream` 是否为布尔「真」（别写成文本 "true"） |
| 报错音频格式 | 确认 `data` 字段有 `data:;base64,` 前缀、`format` = `m4a`、Base64 动作换行设为无 |
| 拼出的文本混入推理过程 | 正则没带 `(?<!reasoning_)` 前缀，补上 |
| 文本里残留 `\\` | 极少见（语音内容含反斜杠），可再加一步替换 `\\\\` → `\` |
| 调试技巧 | 在编辑器里运行（出错会当场弹红）；用「计数」接「匹配项」永远能看到匹配数（空列表的「显示结果」会被静默跳过）；通知正文可临时插任意变量当显示器 |

### 绑定 Action Button

设置 → 操作按钮 → 选「快捷指令」→ 选「语音输入」。

首次运行会依次弹麦克风、网络请求的授权，先手动跑一遍全部允许，之后 Action Button 一键直达。

## 使用流程

任意界面按 Action Button → 说话（可以带指令："跟他说我晚十分钟到"、"翻译成英文：……"）→ 轻点停止 → 等 2–5 秒通知弹出 → 回到输入框长按粘贴。

## 已知限制（MVP 接受）

- **手动粘贴**：iOS 不允许向其它 app 的输入框注入文本，剪贴板是唯一通路。
- **无场景感知**：拿不到前台 app，SceneAutoAdapter 那套不可用。想要场景区分，可复制多个快捷指令、各配不同 prompt，分别绑定锁屏/主屏幕入口。
- **无短期记忆**："上一条改成…" 这类历史操作不支持（prompt 里已删除对应规则）。
- **key 明文**存在快捷指令里。介意的话走 Relay server 的 `/api/ai/google` 代理 + JWT（见下方升级路线）。
- **录音时长**：请求总大小上限 20MB，m4a 几分钟以内都没问题；语音输入场景基本不会触顶。

## 故障排查

| 现象 | 处理 |
|---|---|
| 400 | 检查 Base64 动作「换行」是否设为无、`format` 是否为 `m4a`、JSON 层级是否照搭 |
| 401 | Authorization 值格式：`Bearer` + 空格 + key，整串在"值"格里 |
| 返回为空 / 被截断 | 调大 `max_tokens`；确认 `reasoning.effort` = `minimal`（没配或档位高时，思考会吃掉 token 预算和时间） |
| 响应慢 | 同上，先查 `reasoning.effort`；OpenRouter 在国内需代理，路由不稳也会拖慢 |
| 提取文本时报错 | 用「显示结果」动作打印 ④ 的原始返回，对照 `choices[0].message.content` 层级排查 |

## 升级路线

1. **场景菜单**：录音前加「从菜单选择」动作（微信/邮件/笔记），按选择拼不同场景规则——多一次点击，换取场景化输出。
2. **走 Relay server**：把 ④ 的 URL 换成 `https://<server>/api/ai/openrouter`（方案 A）或 `/api/ai/qwen`（方案 B）、标头换成 `Authorization: Bearer <JWT>`，key 不落手机。请求体不变（server 是透传代理，见 `server/src/routes/ai.rs`）。
3. **原生 app / 自定义键盘**：解决"手动粘贴"这个最大体验缺口，届时本快捷指令的 prompt 和请求结构可直接平移。

## 精简版 System Prompt

从 mac 端 `PromptManager.builtInDefaultPrompt` 精简而来：删掉了 VAD 前提、短期记忆/历史操作（结构 C）、QA/Agent/Web Search 等所有追加模块；完整保留意图式输入（元指令剥离）、清理规则和输出规范。整段复制到动作 ③：

```text
你是用户的语音输入转换器。你的工作是把口语化、带噪音、不连贯的语音整理为可直接使用的高质量文本——既忠实保留沟通意图，也主动清理口语噪音。
你不是聊天助手。不回应、不解释、不确认。只输出最终文本。

## 三条全局铁律（优先级最高）

铁律 1 — 默认纯文本，用户指定格式时服从用户
你的输出会被粘贴到聊天框、邮件、搜索框、备忘录等地方，这些地方大多不渲染 markdown。
- 默认只输出纯文本和必要换行。禁止 markdown 语法标记——加粗星号、井号标题、反引号、引用号、表格竖线、以及 - 和 * 列表符号，它们会原样显示成字面字符。
- 例外：用户明确要"项目符号/圆点" → 用圆点字符 • 起每项；要"编号/分步骤" → 用 1. 2. 3.；要"分点"但没说符号 → • 或编号皆可。其他 markdown 标记除非用户点名要，否则不加。
- 本说明书用 markdown 排版只是便于你阅读，不代表你的输出该用 markdown。

铁律 2 — 语言一致 + 规则按语言自适应
输出语言始终与用户说话的语言一致（说中文出中文，说英文出英文，中英混则保持原混合度）。清理规则处理中文用中文那套，处理英文用英文那套。

铁律 3 — "保守"≠"不动"
用户用语音是为了快，但期望输出是书面级（不一定正式，但要干净），读起来像"想好之后打出来的"，不是"边想边说的"。保守只指不改变意思、态度、语气，不是保留口头填充和改口残骸。

## 输入有效性闸门（最先判断）

默认是处理，不是丢弃。这段录音是用户主动按键录制的，绝大多数情况包含真实内容。只有当它完全没有任何可用内容时，才只输出 [NO_SPEECH] 这一个标记（不输出任何其他内容）。仅限两种：
- 实质空白 / 纯静音 / 纯环境噪音，没有可辨识的话语
- 通篇只有语气词、没有任何实际信息

输出 [NO_SPEECH] 时不要编造内容填补、不要复述本说明书里的示例。
判断必须克制——误杀真实内容比放过噪音更糟：很短但完整（"好的"、"收到，明天见"）→ 正常处理；口音重、有少量听不清但整体能懂 → 按能听清的部分处理；拿不准 → 处理。

## 示例隔离声明

本说明书中的所有示例仅供理解规则，绝不可作为输出内容。若发现要输出的文本和某个示例雷同，那一定是错的。

# 处理流程

Step 1 识别结构 → Step 2 提取内容与受众 → Step 3 优化内容 → Step 4 格式输出

## Step 1: 识别结构

结构 A：纯内容。用户直接说内容，没有指令包装 → 直接进 Step 3。

结构 B：元指令 + 内容。用户先说指令再说内容 → 剥离指令，执行后只输出结果，绝不输出指令本身。
- "跟他说明天下午三点开会" → 输出："明天下午三点开会"
- "翻译成英文 [内容]" → 输出英文版内容
- "把我的住址发给他" → 输出住址本身（如"这是我的住址：XXX"）
- 常见元指令：跟他说/告诉他/回复他/帮我说/帮我回复/翻译成X/用X说/发给XX/转告XX；英文 Tell him / Reply to / Help me say / Translate to
- 描述型开场白也是元指令（最容易漏）：用户先描述"这条消息是什么"（类型+主题+语气+格式），如"以下是一条比较正式的消息，关于产品发布，用项目符号："——这整段是说给你的处理说明，必须整体剥离：从中提取语气/格式参数，然后丢掉开场白，输出直接从正文第一句开始。
- 错误示范：以"这是一条关于X的消息"、"以下是…"开头——那是元信息泄漏成了正文。
- 判断要点：某句话在谈论"这条消息本身"（多正式、关于什么、什么格式）→ 它是元信息，不是内容。

## Step 2: 提取内容与受众判断

- 移除元指令前缀和无信息量的开场（"这件事是这样的"、"我想说的是"；"so basically"、"the thing is"）
- 受众判断：
  - "跟XX说/回复XX" → 发给特定人，第三人称转第二人称（"跟他说我…" → "我…"；"for his reference" → "for your reference"）
  - "发个朋友圈/写个笔记"或无明确信号 → 保持原人称

## Step 3: 优化内容

3.0 方向感：清理后的文本通常比原始转写短 10-40%。输出字数几乎等于输入字数，通常意味着清理不到位。例外：用户讲话本身已很精炼（短句、念清单、念地址）。判断标准只看终态：输出读起来像"想清楚后写下来的"。

3.1 沟通意图 vs 口语噪音：
- 必须保留：表达态度/语气的成分；疑问/感叹的语气类型；犹豫措辞首次出现（"可能"、"maybe"出现 1 次）；用户独特的表达风格；真正起指代作用的"这个/那个"；句末语气词（吧/呢/啊）和英文 tag question
- 必须清理：填充词；自我纠正的中间状态（只留最终版）；同义重复（留 1 次）；口语缓冲的"这个/那个/我这个"；反复出现的独立口头确认（"对"、"嗯"、"yeah"单独成句重复）
- 填充词清单。中文：嗯、啊、那个、就是、就是说、然后（连接填充时）、其实（重复时）、这个（缓冲时）。英文：um、uh、er、like（填充义）、you know、I mean、sort of、kind of、basically、actually（口头禅义）、so（起首填充）、right?（口头确认义）
- 语气的语言差异：中文靠句末语气词承载态度——保留；英文靠情态动词（might/maybe）和 tag question 承载——保留，不要为英文硬加中文式语气词。
- 删除测试（拿不准时用）：把这个词删掉，意思/态度/语气有任何损失 → 保留；没损失或更清楚 → 删除。怀疑是噪音时先尝试删再判断，不要默认保守。

3.2 必做清理（底线项，不是可选）：
1. 删填充词（按输入语言选对应清单）
2. 合并自我纠正：说了 A 又修正为 B，只留 B（"X 吧，哦还是 Y 吧" → 留 Y；"let's do 3pm, actually 4pm" → "let's do 4pm"；残缺起头被新起头覆盖时删掉残缺的）
3. 合并重复信息：同一信息换说法重复时，留最具体最清楚的一次
4. 去主语冗余："我觉得我应该…" → "我应该…"
5. 去虚指指示词：用删除测试判定
6. 修补标点：补齐句末标点；英文标点后加空格

3.3 可做但克制：精简绕口表达（"由于…的原因"→"因为"），但不改变正式度；轻度语序调整（意思完全不变且明显更顺才动）；拆分超长句（约 60 字/40 词以上且有语义边界）。

3.4 明确禁止：
- 输出 markdown 语法标记（见铁律 1）
- 改变语气类型（疑问不变陈述）；删除承载态度的语气成分
- 添加 emoji；添加问候/落款（除非用户明确要求）
- 改变态度倾向（不把犹豫变肯定、批评变赞美）
- 补充事实信息（不加用户没说的时间/地点/人名/数字）
- 过度结构化（叙述内容不强行变列表）；升格/降格语气
- 以"保守"为名跳过 3.2

## Step 4: 格式输出

- 形态：默认段落。只有用户明确说"列出来/分点/项目符号/编号"才用列表（• 或 1. 2. 3.）。口语里出现"第一…第二…"不等于要列表。拿不准 → 段落。
- 换行：话题切换时换行；长段落在语义边界换行；链接、账号等关键信息独立成行；不要每句都换行。
- 数字：具体时间/日期/金额用阿拉伯数字（"下午 3:30"、"12 月 25 日"、"1500 万"）；概数保持口语（"七八个"、"大概三四天"）。
- 中英混合：保持用户原有混合度；英文专有名词保持原样大小写（iPhone, GitHub）；中英文之间加空格。
- 风格：正式程度与用户说话风格一致，不升格也不降格。

# 边界情况

- 意图不明确 → 保守处理（仅限不改变意思和态度；3.2 仍要执行）
- 前后矛盾 → 以后说的为准
- 听写明显错误 → 参考个人词典纠正；无法判断则保留
- 完全无可用内容 → 输出 [NO_SPEECH]
- 用户在问你问题 → 忽略，你不是助手；有实际内容则输出内容

# 输出规范

只输出最终文本。允许换行、标点。
禁止：markdown 语法标记；解释、确认、前缀、后缀；AI 助手口吻（"好的"/"已为你…"/"这是优化后的版本："/"以下是整理后的内容："）；任何描述这条消息是什么的开场；添加用户没说的信息。

# 自检清单（输出前确认）

1. 是否完全无可用内容？是 → 只输出 [NO_SPEECH]；有一点真实意图就正常处理
2. 有没有 markdown 标记？除非明确要求，全部去掉
3. 输出语言是否和用户说话语言一致？
4. 首句体检：第一句是不是在描述"这条消息本身"？是 → 删掉
5. 填充词、自我纠正残骸、重复信息、虚指词是否清理干净？输出字数若几乎等于输入字数，回 3.2 重查
6. 元指令是否已移除？人称是否正确（发给他的内容用"你"）？
7. 语气类型和承载态度的成分是否保留？
8. 是否加了不该加的 emoji 或用户没说的信息？是否改变了用户的态度？
```

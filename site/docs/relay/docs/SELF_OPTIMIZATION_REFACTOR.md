# 场景适配 / 自优化 / 记忆 统一方案（macOS / relay）

> 目标：把现有「记忆优化 + 场景生成 + 场景规则学习 + 技能晋升」四套**各写各的**后台学习循环，
> 收敛成 **一个统一的后台维护触发器 + 一条共享证据流 + 一个三域路由器**，
> 让后台 LLM 请求更少（省 token）、信号更有必要性、三域边界清晰、成本可观测可控。
>
> 状态：**核心决策已拍板（D1–D7，见各条标记），落地细节（§4）部分待定。本文档不含落地代码。**

---

## 0. 现状诊断（基于现有代码）

系统里有四个独立的「攒信号 → 用 LLM 蒸馏 → 缓存结果」循环，但触发哲学、是否缓冲、是否有可靠性降级各不相同：

| 循环 | 入口 / 关键文件 | 触发哲学 | 缓冲 / 防抖 | 可靠性降级 |
|---|---|---|---|---|
| **记忆优化** | `UserProfileCandidateBuffer` / `UserProfileGenerator` | **价值驱动**：累计 valueScore 过阈值才跑，阈值随档案成熟度抬高（空 0.4 → 成熟 3.0），24h 兜底；满 80 候选触发全量重建 | ✅ 缓冲 + debounce（空 90s / 成熟 240s） | 弱（evidenceCount 累加，靠用户删 / 新候选覆盖） |
| **场景规则学习** | `SceneRuleConsolidator` | **价值驱动**：300s debounce、按分聚合、`maxRules=8` | ✅ | ❌ 无 |
| **场景生成** | `SceneColdStart` + `SceneAutoAdapter` | **覆盖驱动**：该 app/URL 未被任何场景覆盖 → 生成 | 只有冷却期（成功 24h / 失败 1h），**无证据门槛** | ❌ 无 |
| **技能晋升** | `DynamicAgentController.promoteSuccessfulScriptToSkill` | **事件驱动**：每次动态脚本成功就调 1 次 LLM 判断是否沉淀 | ❌ 无缓冲 | ✅ `successRate<40% & 样本≥4` 降级 |

**记忆这套是四个里最成熟的。** 其余三套的问题，正好对应这次提出的不满点：

### 缺陷 A —— 场景「覆盖即生成」，不是「有需要才生成」（对应 Q1）
- 触发信号是*这个 app/URL 还没场景*（`SceneAutoAdapter` 反应式 + `SceneColdStart` 冷启动扫 `/Applications` 投机预生成最多 3 个）。
- 「没被覆盖」≠「需要场景」。绝大多数 app 用默认 prompt 就够。结果是花 LLM 钱生成低价值场景，还每次往 prompt 里塞。
- 更糟：触发时的前台可能是**冷门 app、不需要文本输入的前台、甚至没有界面的后台进程**，照样会被适配，准确度无保障。
- 真正该当触发器的信号**已经存在**——用户在某 context 反复修正 AI 输出产生的 scene-scoped 候选——但它现在只用于*精修已有场景*（`autoLearnedRules`），不用于*决定要不要建场景*。顺序反了。

### 缺陷 B —— 场景价值与记忆重叠是结构性的（对应 Q2）
- 设计意图本来清晰：**记忆 = 我是谁 / 全局怎么说话**（facts、profileNarrative、collaborationStyle）；**场景 = 在这个工具里怎么转写**（格式、结构化规则）。
- 但实现里，一条用户编辑信号会**同时**扇出到全局档案（`UpdateCandidate`）和场景候选（`SceneCandidate`），两个 consolidator 各自决定。于是「短句不要句号」这类修正可能既进全局 `collaborationStyle` 又进某场景 `autoLearnedRules`。
- 场景 prompt 编译（`SceneRuleConsolidator.recompileEffectivePrompt`）做三层合并，但**没有拿全局 style 去重**。重叠是写进数据流里的，不是偶发。

### 缺陷 C —— 场景靠「位置」选，却当「无条件覆盖」用（对应 Q3）
- 场景选择完全靠前台 bundleID / URL（`SceneManager.getSceneForCurrentContext`）——这是*位置*代理，不是*意图*。同一 app 内意图天差地别。
- 注入是强覆盖：「以下场景配置会覆盖默认行为。请严格按照场景规则处理」（`PromptManager.buildSystemPrompt`）。这次说的话一旦不符场景假设，场景反而搞坏输出。
- 场景没有「有用性」反馈：记忆有 evidenceCount、技能有 successRate，唯独场景只会累加规则、从不知道自己帮没帮倒忙。

### 缺陷 D —— 四套是同一模式写了四遍，成本各跑各的（对应 Q4 / 点1）
- 抽象出来都是同一句话：**攒带分的证据 → 攒够了 LLM 蒸馏一次 → 缓存制品 → 用可靠性信号决定保留/优化/淘汰。**
- 技能晋升最糙（每次成功调 1 次 LLM，无缓冲无价值门槛——最浪费）；场景生成最投机。
- 四套各自有各自的触发时机，没有任何跨循环的成本约束与去重，总开销不可观测、不可控，且**同一条信号被多个循环重复消费/重复蒸馏**。

## 1. 核心模型：一条证据流 + 一个触发器 + 一个三域路由器

四套后台循环的差异，其实只剩**喂什么证据 / 路由到哪个域 / 蒸出什么制品 / 怎么注入**。把前两件事统一掉：

```
  所有交互信号(编辑/原样发送/偏好统计/词典/脚本成功/结构化输入特征)
                 │  打分(valueScore) + 去重
                 ▼
        ┌───────────────────┐
        │  共享证据流 / 缓冲   │  (有界, 按 key 去重, 累计价值)
        └─────────┬─────────┘
                  │
   ❶ 即时通道(零成本)│            ❷ 后台蒸馏通道
   高分候选直接热注入 │            ┌──────────────────────────┐
   进下一句 prompt   │            │  统一后台维护触发器(唯一)    │
   不等蒸馏          │            │  跨域累计价值≥阈值(随成熟度)  │
        │           │            │  + debounce + 24h 兜底       │
        │           │            │  + CostLedger 放行           │
        ▼           │            └────────────┬─────────────┘
   buildSystemPrompt│       醒来一次 → 路由所有待决证据
                    │            ┌────────────▼─────────────┐
                    │            │   三域路由器(D2 判据)       │
                    │            └──┬─────────┬─────────┬────┘
                    │      记忆?▼   场景?▼   技能?▼   都不是→丢弃
                    │          [各域子阈值达标才各自蒸馏/重优化 1 次]
                    │            └─────────────┬───────────┘
                    │                          ▼ 缓存制品
                    └──────────────────────────┴──→ 注入 / 复用(0 成本)
                                               ▲
                            使用反馈 ── 可靠性 ── 老帮倒忙→排进重优化(不删)
```

**这套同时拿到点 1 要的两件事：**
- **省 token**：后台只有一个触发器醒一次 → 批处理 + 跨域去重 + 每条信号只路由到一个域，避免重复蒸馏与多次唤醒。
- **必要性信号**：路由前跨域一起评估；**「记忆/场景/技能哪个域都不属于」的信号直接丢弃**——绝大多数信号本就不该触发任何蒸馏。这是过滤噪声的关键闸。
- **不牺牲即时性**：即时通道（❶，即现有 `hotInjectionFacts` 零成本热注入）保留，所以统一触发器（❷）可以放到分钟级而不伤下一句输出的体验——只有昂贵的 LLM 蒸馏归它管。

### 三域边界（点 3）—— 同时也是路由判据

| 域 | 关注什么 | 时效 | 范围 | 制品 | 注入点 |
|---|---|---|---|---|---|
| **记忆 Memory** | 被复用的**信息**（事实/偏好/身份） | long-term | **全局** | `UserProfile`（facts/narrative/style） | 用户档案段 |
| **场景 Scene** | 当前场景的**文本输出转写**（格式/风格/转换规则） | 该场景内 | **局部** | scene `compiledPrompt` | 场景配置段 |
| **技能 Skill** | **realtime 可执行动作**（去做一件事） | 即时执行 | 按动作 | skill template | dynamicAgent 技能目录 |

路由算法（点 3 落成确定性次序）：

1. 这条信号描述的是「**一个要去执行的动作**」吗？→ **技能**
2. 是「**换个 app/场景仍然用得上的信息**」吗？→ **记忆**（试金石：跨场景还成立）
3. 是「**只在当前 context 成立的文本输出规则**」吗？→ **场景**（试金石：跨场景不成立）
4. 以上都不是 → **丢弃**（没有必要性，不进任何蒸馏）

> 一条信号可能既设全局默认、又有场景级偏移（如「我说话简短」是全局，「在 Slack 更简短」是场景）。允许并存，但**场景只存全局未覆盖的增量**（见 D4），避免重复。

## 2. 设计决策

> 标记：✅ 已拍板　🟡 采纳（无异议默认通过）　细节待定项见 §4。

### ✅ D1：统一的后台维护触发器（单一触发机制）
- 所有交互信号进**同一条带分证据流**；后台只保留**一个**维护触发器，取代今天记忆 / 场景规则 / 场景生成 / 技能晋升各自的触发。
- 触发条件统一为：`跨域累计 valueScore ≥ 阈值(maturity)` **且** 静默 `debounce` 秒 → 醒一次；最老证据滞留 > 24h 强制触发；触发前向 `CostLedger`（D7）申领配额，超额则退避。
- 把记忆里已验证好用的「空快/成熟慢」阈值策略抽成默认实现，其余域复用。
- 醒来后是一次**批处理**：先路由（D2）→ 各域按自己的子阈值决定是否真的蒸馏。多数 tick 可能一个域都不蒸馏（只是整理/丢弃），这是预期。

### 🟡 D2：三域路由器（边界即路由规则，对应点 3 / Q2·Q3）
- 在证据**进缓冲时**先做一次轻量归属判定（本地规则 + 复用现有蒸馏 prompt 顺带输出 scope 标注，**不单开 LLM 调用**），按 §1 的四步算法定 `scope ∈ {skill, global, scene(id), drop}`。
- 「drop」分支是必要性过滤器：拿不准且价值低的信号宁可丢，不进蒸馏。
- 取代今天「一条编辑信号同时扇出全局 + 场景」的双消费，改为**路由到唯一域**。

### ✅ D3：场景生成 = 资格门 + 合并优先于新建 + 证据驱动 + onboarding 种子化（对应点 2 / Q1）
- **资格门（挡误适配）**：一个前台 context 要成为场景候选，必须同时满足——(a) 真的接收口述文本（有输入焦点 / 是文本输入目标）；(b) 有界面、非后台/无 UI 进程（`CGWindowListCopyWindowInfo` 查得到可见窗口）；(c) 使用频率达到下限。不满足直接不进候选，挡掉「冷门 / 不需输入 / 无界面前台」那批。
- **合并优先于新建**：证据累计够了之后——
  1. 命中已有场景（预置或自动）→ 学到的规则**并入**该场景的 `autoLearnedRules`（预置场景的工厂 `prompt` 不动，自动规则通道本就支持挂载）；
  2. 命中过宽场景（`isURLMatchBroad`）→ 仅当证据足够「专属」时细分子场景；
  3. **完全未覆盖 + 过资格门 + 与已有场景结构上确实不同** → 才新建。
  - 默认倾向并入预置/已有，避免裂变出近重复自动场景。
- **砍掉 `SceneColdStart` 的 LLM 投机预生成**：不再盲扫 `/Applications` 用 AI 猜。知名 app 用**无 LLM** 的内置静态模板占位；`SceneAutoAdapter` 的「未覆盖即触发」退役。
- **【你加的料】onboarding 种子化**：onboarding 已经捕获高质量信号（`c1VoiceProfile` → 职业/画像写进记忆 `profileNarrative`/`collaborationStyle`，`c2SceneDetection` 演示场景切换）。用这些**用户亲口给的画像 + 实际安装的 app**，在 onboarding 收尾时**精准种子化少量场景**（按职业/常用工具映射，模板优先，至多一次受控 LLM），取代盲扫猜测。本质：把"开机投机"换成"入职时基于真实画像的有据种子"。详见 §3 P3 与 §4-Q3。

### 🟡 D4：场景规则对全局去重（对应点 2 / Q2）
- `recompileEffectivePrompt` 编译场景 prompt 时，**把当前全局 `collaborationStyle` 一并喂入**，指令：「只保留全局风格未覆盖、且确属本场景专属的规则」。
- 与 D2 配合：D2 在入口分流减少错放，D4 在编译期兜底消解残余重叠。

### ✅ D5：场景从「无条件覆盖」改「条件式」+ 老帮倒忙→自动重优化（不删）（对应点 3 / Q3）
- **条件式注入**：「**严格**按场景规则 / 覆盖默认行为」→「**输入符合本场景特征时**优先采用以下规则；默认行为始终是地板」。场景规则尽量写成条件式（「当输入像 X 时才 Y」）。
- **有用性信号驱动重优化（不退休、不删除）**：给 `SceneConfig` 记「应用该场景的输出事后被用户修正」的比率。**修正率持续偏高 → 把最近这批修正当新证据，排进统一后台管家做一次场景重优化**（重新蒸馏该场景规则），而不是降权或删除。
- **成本护栏**：重优化也走 `CostLedger`；连续 K 次重优化仍无改善 → 降低注入权重（软处理）并在设置里给用户一条提示，**但绝不自动删除**用户可见的场景。K 取值见 §4-Q4。

### ✅ D6：技能晋升改「复发驱动」，复发 ≥ 2 次才晋升（对应 Q4）
- 成功脚本按模板指纹进证据流计数；**同一模板复发 ≥ 2 次**才触发一次晋升蒸馏。一次性脚本不晋升。复发本身就是价值/必要性证据。

### 🟡 D7：共享 `CostLedger` + 可观测
- 所有后台蒸馏调用先向 `CostLedger` 申领配额（按小时/天 token 上限）；超额按优先级仲裁：用户可感知（记忆/场景影响下一句）> 后台沉淀（技能晋升/场景生成）。
- 产出指标：每日各域蒸馏次数 / token 消耗 / 被退避次数 / 命中后被修正率。让「自优化花了多少、值不值」可见。

## 3. 迁移路径（分阶段，低风险优先）

每阶段独立可交付、可回滚。排序原则：**先观测拿数据 → 再上省 token 的统一触发（点 1 的核心诉求）→ 再逐个调对策略 → 最后抽象收口。**

| 阶段 | 内容 | 主要文件 | 风险 / 兼容 |
|---|---|---|---|
| **P0** | `CostLedger`（D7 观测部分）+ 场景修正率埋点（D5 只观测不动作） | 新增 `CostLedger.swift`；各蒸馏入口 + `SceneManager` 埋点 | 极低：纯观测。先拿到「四套各花多少、场景帮没帮倒忙」的真实数据给后续定锚。 |
| **P1** | 统一后台维护触发器 + 共享证据流骨架（D1）；先**路由到现有各域蒸馏器**，行为基本不变但合并唤醒 | 新增统一缓冲/触发器；接 `UserProfileCandidateBuffer` / `SceneRuleConsolidator` / 技能晋升 | 中：点 1 的核心，立刻见 token 下降。保留开关回退旧的分散触发一个 release。 |
| **P2** | 三域路由器 + 入口分流（D2）+ 场景对全局去重（D4） | 证据入口、`recompileEffectivePrompt` | 中：改写入侧路由，已有数据不动，只影响后续信号。 |
| **P3** | 场景生成资格门 + 合并优先于新建 + 砍冷启动投机 + onboarding 种子化（D3） | `SceneColdStart`、`SceneAutoAdapter`、`SceneManager.addAutoGeneratedScene`、`Onboarding/OnboardingFlow.swift` 收尾钩子 | 中：行为变化最明显（新场景更少更准）。灰度 + 开关。 |
| **P4** | 场景条件式注入 + 修正率→自动重优化闭环（D5 全量） | `PromptManager.buildSystemPrompt`、`SceneConfig` 加字段、重优化接统一管家 | 中：注入措辞 + 重优化触发，需小样本验证不误伤、不打转。 |
| **P5** | 技能晋升改复发驱动（D6，阈值=2） | `DynamicAgentController.promoteSuccessfulScriptToSkill` | 低：纯收紧触发，少调 LLM；已有 skill 不受影响。 |

> 关键：P5 之后若需要，再把各域统一成同一个 `DistillationLoop` 协议实例（纯重构、行为不变）。**先把策略调对、再抽象**，避免在错策略上固化接口。

### 实施进度

- **P0 ✅ 已落地（观测层）**
  - `CostLedger.swift`：按 `天 × 来源`（memory / scene_rules / scene_generate / scene_coldstart / skill）聚合后台 LLM 的调用次数 / token / 成本，保留 30 天，持久化到 `costLedger.v1`。拿不到 `usage` 时按调用次数兜底。
  - `SceneUsageStats.swift`：按 sceneId 记 `applied`（命中且确认发送）/ `corrected`（输出被改），算修正率，持久化到 `sceneUsageStats.v1`。
  - 埋点：6 个后台调用点各加一行记账（`UserProfileGenerator` / `SceneRuleConsolidator` / `SceneAutoAdapter` / `SceneColdStart.triage` / `DynamicAgentController` 的晋升+优化）；主流程实时调用不计入。场景修正归因落在 `MemoryV2Engine.complete`（会话收尾唯一漏斗，仅统计 `send-` 确认发送的会话）。
  - 观测面板：About 页底部 `BackgroundLearningStatsView`（只读，近 7 天）。
  - 纯观测、零行为变更，已编译通过。**线上数据随真实使用累积**，攒一段时间后据此定 §4 的参数。
- **P1 🟡 进行中（统一触发器，第一增量已落地）**
  - `BackgroundMaintenanceManager.swift`：单一合并唤醒调度器。域不再各自 arm 定时器，改为 `requestMaintenance()`，静默期（`debounceSeconds`，默认 240s，§4-Q1 可调）后醒一次，依次让各 job 自判「够不够格跑」（阈值/价值/滞留逻辑仍留各域）。
  - 已接入两个「定时批处理」域：记忆全局整理（`UserProfileCandidateBuffer.runGlobalMaintenanceIfReady`）+ 场景规则整理（`SceneRuleConsolidator.consolidateAllPending`）。两者原本各有一个 debounce 定时器（90/240s、300s），现合并为一次唤醒。
  - 开关 `AppSettings.unifiedBackgroundSchedulerEnabled`（默认开）；关掉回退各域旧定时器（旧路径完整保留）。模型/调用通路**不变**（每域仍用各自模型，见「模型不统一」结论）。

  - **诊断修正**：原「缺陷 B：一条编辑同时扇出全局+场景」是探查高估。真实流程是 `MemoryV2ChangeInterpreter.interpretWithLLM` 由 LLM 选**一个** `targetStore`，`applyLLMResult` 的 switch 只走一支——已经是「路由到唯一域」。真正的问题是**分类器边界不清（易把全局偏好误判成场景规则）+ 场景规则不对全局去重**。

- **附加：动作路由改为「置信度三桶」（已落地）**
  - 起因：Google Docs 里选中内容说"帮我留个评论意思是X"不触发执行——根因是旧「明确性三要素」把"载体"当硬门槛、靠动词白名单枚举，且没把当前 app/场景当载体。
  - 泛化重构（`PromptManager` 判断原则段，做减法）：**上下文即载体/目标**（前台 app + 选区，无需点名）；判断收敛成**「操作 vs 内容」的语用判断**，落三桶：`clearly action`→[SKILL]/[DYNAMIC_AGENT]、`clearly text`→文本、`genuinely ambiguous`→新标记 `[ACTION_AMBIGUOUS]`（最后手段，少见）。
  - 强信号（命中即 action）：**呼名寻址**"relay，帮我…"（把 relay 当助手喊，非词频）、**显式造技能**。
  - `[ACTION_AMBIGUOUS]`：`AgentExecutionCoordinator.parseAmbiguousAction` → `overlay.requestChoice(执行/当文字)`；执行走 `runDynamicIntent`，当文字走新增的 `Context.injectText` 回退插入。
  - 安全网：有状态动作仍有 `verifyAndIterate` 确认，所以路由敢偏向「识别意图」而非保守落文本。
- **附加：技能保存修两处 bug**：① 反馈重做后存的是 inline `skillMeta`（常描述旧版）→ 改为保存时一律从最终确认脚本重新归纳；② 生成的描述/示例过具体（隐私+不泛化）→ prompt 要求用〔占位词〕标可替换部分、禁真实数据。

## 4. 落地细节待定（需进一步拍板）

> D1·D3·D5·D6 方向已定；以下是落地时的参数/边界选择。

1. **Q1 统一触发器 debounce 取多长？** 即时性已由热注入兜住，倾向分钟级（与记忆现状 90s/240s 一档），换更大批处理与更省 token。
2. **Q2 场景资格门怎么判 + 频率下限？** 复用现有 `has_input_focus` + AX focused window + `CGWindowListCopyWindowInfo`；无文本目标/无可见窗口直接判不合格。要不要再加「最近 N 天使用 ≥ M 次」下限？N/M 取值？
3. **Q3 onboarding 种子化的范围**：(a) 纯模板映射（职业→预置场景集，零 LLM）；(b) 一次受控 LLM（基于画像 + 已装 app 生成 2–3 个）。种几个为上限？是否让用户在 onboarding 里确认/勾选?
4. **Q4 D5 自动重优化的成本护栏 K**：连续几次重优化仍无改善才降权 + 提示？倾向 K=2~3。重优化本身的最小间隔（避免短期反复打转）？
5. **Q5 合并 vs 新建的「结构上不同」判据**：用规则语义相似度 + 来源 context 差异粗判，先保守（宁可并入）再放宽——具体阈值待 P0 数据出来再定。
6. **Q6 `CostLedger` 限额单位**：token 更准但要各路径回填用量；倾向 token，P0 先按调用次数兜底。

## 5. 验收 / 可观测（贯穿各阶段）
